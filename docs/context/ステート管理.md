# context/

このフォルダは、English Conversation AI アプリケーションのグローバルな状態管理をまとめたものです。

## 目的

- モード・会話履歴・Exam 状態・ペルソナ・API ローディングなど、アプリ全体で共有する状態を一元管理する。
- React Context を通じてコンポーネントやページに単一の信頼できる状態ソースを提供する。
- Reducer により状態遷移のルールを明確化し、UI コンポーネントは表示ロジックに集中できるようにする。

## ファイル

- `AppContext.tsx` — Provider、reducer、アクション定義、初期状態、`useApp()` フックを含む。

## 状態の形（必須）

アプリコンテキストは次のフィールドを提供する必要があります（型は `types/index.ts` を参照）：

- `mode: AppMode` — `"normal"` または `"exam"`。
- `messages: Message[]` — 通常モード用の会話履歴（古い順→新しい順）。
- `examMessages: Message[]` — Exam モード用の会話履歴（古い順→新しい順）。
- `examRallyCount: number` — Exam モードの現在のラリー数（0〜10）。
- `examResult: ExamResult | null` — Exam 終了後の結果（スコアとフィードバック）、未評価時は `null`。
- `persona: PersonaSettings` — AI に渡すシステムプロンプト／人格設定。
- `isLoading: boolean` — API 通信中フラグ（グローバル）。

セッション関連（推奨）:

- `sessions: Record<string, Session>` または `Session[]` — 保存されたセッション群（各セッションは `messages` / `examMessages` 等を含む）。
- `activeSessionId: string | null` — 現在操作中のセッションを指す識別子。

要件:

- 配列やオブジェクトは不変に扱う（reducer は新しい配列／オブジェクトを返す）。
- `examRallyCount` は 10 を超えないように制約を設けること（reducer 側での検証を推奨）。
- `examResult` は Exam 実行前は `null` を維持すること。

## サポートするアクション（必須）

reducer は TypeScript の識別共用体（discriminated union）で以下のアクションを扱えること：

- `SET_MODE` — payload: `AppMode`。`exam` に切り替える場合は Exam 用状態をリセットすること。
- `ADD_MESSAGE` — payload: `Message` — `messages` に追加する。
- `ADD_EXAM_MESSAGE` — payload: `Message` — `examMessages` に追加する。
- `INCREMENT_RALLY` — payload: なし — `examRallyCount` を +1（最大10で上限）。
- `SET_EXAM_RESULT` — payload: `ExamResult` — `examResult` を設定する。
- `SET_PERSONA` — payload: `PersonaSettings` — `persona` を更新する。
- `SET_LOADING` — payload: `boolean` — `isLoading` を設定する。
- `RESET_EXAM` — payload: なし — `examMessages` を空にし、`examRallyCount` を 0、`examResult` を `null` にする。

追加アクション（セッション管理）:

- `CREATE_SESSION` — payload: `{ id: string; name?: string; mode?: AppMode }` — 新しいセッションを作成して `activeSessionId` を切替える。
- `SWITCH_SESSION` — payload: `{ id: string }` — `activeSessionId` を切替え、対応セッションの状態を読み込む。
- `DELETE_SESSION` — payload: `{ id: string }` — セッションを削除する。
- `SAVE_SESSION` / `LOAD_SESSION` — 永続化・復元操作をトリガーする（実装は副作用側で扱う）。

実装上の注意:

- アクションは型安全に定義すること（TypeScript の利点を活かす）。
- reducer 内で直接 state を変更しないこと（不変性を保つ）。
- API 呼び出しや永続化など副作用は reducer に入れず、コンポーネントやフック側で扱うこと。

## Provider / Hook の API（要件）

- `AppProvider` コンポーネントをエクスポートし、アプリ全体をラップして `{ state, dispatch }` を提供すること。
- `useApp()` フックをエクスポートし、Provider 外で使われた場合はエラーを投げること。
- コンポーネント側は定義されたアクションのみを `dispatch` で送ること。必要ならばアクション作成ヘルパーを用意する。

## 永続化とリセット

- `persona` などの非機密な設定をローカルストレージに任意保存する将来の拡張を想定。ただし永続化ロジックは reducer 外で実装すること。
- Exam 状態のリセットは単一のアクション（`RESET_EXAM`）で行えるようにする。

- セッション永続化: `sessions` の保存／復元はコンテキスト外（サービス層）で行い、保存／読み込みの完了時に `LOAD_SESSION` 相当のアクションで差し替える設計を推奨する。

## 同時実行性とローディング

- `isLoading` はグローバルな API 通信用フラグ。API 呼び出しの開始時に `true`、完了時に `false` にする。
- 同時に複数リクエストが発生する場合は、単一の boolean が不十分なことがあるため、将来的にリクエストカウンタ方式やリクエスト毎のローディング状態を検討する。

## テスト

- reducer の各アクションに対する動作（ラリー数の上限、Exam リセットなどのエッジケースを含む）を単体テストで検証すること。
- `useApp()` が Provider 外で呼ばれた場合に例外を投げることをテストすること。

## 今後の検討事項

- 状態管理が複雑化した場合は `useReducer` から Redux / Zustand / Jotai 等への移行を検討する。
- API 連携を整理するためのアクション作成関数や非同期ヘルパー（thunk / saga）を導入する。
- `persona` 等の選択的な永続化（localStorage）を追加する。

## 使い方（簡易）

ルートを `AppProvider` でラップし（`pages/index.tsx` ですでに行っている）、コンポーネント内で次のように利用します：

```tsx
const { state, dispatch } = useApp();
```

必要に応じて `dispatch({ type: 'ADD_MESSAGE', payload: message })` のようにアクションを送信してください。
