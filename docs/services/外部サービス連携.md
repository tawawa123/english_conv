# services/

このフォルダは、外部サービスやブラウザ API との通信を抽象化するユーティリティを収めます。
アプリケーションの UI 層（`components/` や `pages/`）から呼び出され、API キーやリトライ等の環境依存ロジックを隔離します。

## 目的

- STT（音声認識）・TTS（音声合成）・AI クライアント等、外部サービスを呼び出す責務を集中させる。
- コンポーネント側に副作用や設定の詳細を漏らさず、テストしやすいインターフェースを提供する。
- エラー処理、タイムアウト、リトライ、ログ記録、API キー管理などを一元化する。

## 想定ファイル

- `stt.ts` — ブラウザの Web Speech API（あるいは外部 STT）をラップする。`start(onResult, onEnd)` / `stop()` のようなシンプルな公開 API を提供。
- `tts.ts` — ブラウザの SpeechSynthesis（あるいは外部 TTS）をラップする。`speak(text, options)` / `stop()` / `isSpeaking()` 等を提供。
- `aiClient.ts` または `claudeClient.ts` — AI モデル（Claude, OpenAI 等）への呼び出しを行う。`sendMessages(messages, personaPrompt)` などの関数を提供し、レスポンス整形とエラーハンドリングを行う。
- `supportService.ts` — `suggest` や `ask` といったサポート系 API のクライアントロジック（`pages/api/support.ts` と連携する場合は、クライアント側のラッパーを配置する）。

## 設計と要件

- 非同期 API はすべて `Promise` を返すこと。
- タイムアウト: 長時間待たせないためのタイムアウトを組み込むこと（デフォルト 15〜30 秒程度を推奨）。
- リトライ: 一時的なネットワーク障害に備えて指数バックオフでのリトライを検討する（ただし重複実行の影響に注意）。
- キャンセル: ユーザーが操作をキャンセルできるように `AbortController` をサポートすること。
- エラーハンドリング: 呼び出し元が扱いやすい形式（`{ ok: boolean, value?: T, error?: string }` など）で返すか、適切な例外を投げる。
- ログ: 重要なエラーや API レイテンシはコンソールや外部ロギングに記録する（機密情報は除外）。
- 設定: API エンドポイントやキーは環境変数（`.env` / Vercel 環境変数）で管理すること。

## `stt.ts` 詳細要件

- ブラウザの `SpeechRecognition`（Web Speech API）を利用する想定だが、未対応ブラウザではフェールオーバー（テキスト入力のみ）を行う。
- API: `start(onResult: (transcript: string) => void, onEnd?: () => void)` / `stop()`。
- マイク許可の取り扱い: 権限拒否や未サポート時にわかりやすいエラーを返す。
- 複数回の start/stop に耐える実装（再入可能）。
- テスト向けにモック実装が容易な設計にする。

## `tts.ts` 詳細要件

- ブラウザの `SpeechSynthesis` を利用する想定。外部 TTS サービスを使う場合はエンドポイントラッパーを追加。
- API: `speak(text: string, opts?: { voice?: string; rate?: number; pitch?: number; lang?: string }): Promise<void>` / `stop(): void` / `isSpeaking(): boolean`。
- キュー管理: 連続した `speak` 呼び出しに対してキューまたは上書きの戦略を持たせる。
- SSR 安全: サーバーサイドレンダリング環境では何もしないようにガードする。

## `aiClient.ts` / モデル呼び出しの要件

- 入力: `messages: ApiMessage[]`, `personaPrompt?: string` を受け取る関数を提供する。
- 出力: 正常時は `{ reply: string }` などの整形済みレスポンスを返す。
- ストリーミング対応: 将来の拡張としてストリーミング応答（部分応答を逐次表示）をサポートする設計にする。
- レート制限とエラー: API のレート制限エラーを判別し、呼び出し元に明示的に返す。
- テスト: モック可能なインターフェースを提供し、ユニットテストで AI 呼び出しをシミュレートできること。

セッション永続化サービス（推奨）:

- `sessionService.ts` — クライアント側のセッション保存／読み込みの責務を持つ。ローカルストレージやサーバー API（`/api/sessions`）を使って `sessions` の永続化を担う。
- 永続化は `Context` の副作用（フック）から呼び出し、保存完了時に `LOAD_SESSION` 相当のアクションで状態を復元する。

## サポート系サービス

- `supportService` は `suggest(context)` と `ask(question)` のような高レベル関数を提供し、サーバーサイドの `pages/api/support.ts` と通信する。
- クライアント側の呼び出しはシンプルで、UI は期待される配列や文字列を直接描画できる形式で受け取る。

## 環境変数

- `AI_API_KEY`（例: Claude / OpenAI のキー） — サーバーサイドで使用。
- `AI_API_URL` — カスタムエンドポイントを利用する場合。
- その他外部 STT/TTS サービスを使う場合は個別にキーを定義する。

## セキュリティとプライバシー

- API キーや機密データはクライアントバンドルに含めないこと。
- ユーザーの音声データや会話ログを保存する場合は明示的な同意が必要。保存する場合は暗号化やアクセス制御を検討する。

## テストとモック

- 各サービスモジュールはインターフェースを抽象化し、ユニットテストではモック実装を注入して副作用をシミュレートする。
- `stt` / `tts` のブラウザ API は Jest や Playwright のモックで置き換え可能な形にする。

## 使い方の例（クライアント側）

```ts
// 会話送信前に STT を止める
sttService.stop();

// AI に送信
const res = await aiClient.sendMessages(messages, personaPrompt);
if (res.reply) {
  // UI に反映
}
```

## 将来的な拡張

- 外部音声サービス（Google, Azure, Amazon 等）の追加対応。
- ストリーミング音声認識 / 合成のサポート。
- サーバーサイドでのバッチ処理やキューを導入して大きな処理を分割する。
